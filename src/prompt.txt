Prepare, deploy, test, and review my personal website, **mathlog**.

The project currently runs on localhost. Deploy it to **Vercel**, using a Preview deployment first and Production only after testing.

1. Inspect the project

Review the project structure, package scripts, framework, environment variables, and existing deployment configuration.

Use the package manager specified by the lockfile. Do not change the architecture or add unnecessary dependencies.

2. Run and clean up the project

Run the website locally and execute the available checks, such as:

* Linting.
* Type checking.
* Tests.
* Production build.

Fix any errors that affect the website.

Remove clearly redundant parts, including:

* Unused files, imports, styles, assets, and dependencies.
* Duplicate components or repeated code.
* Dead routes.
* Duplicate content or unnecessary UI elements.
* Debugging code and logs.

Do not remove anything that has a distinct visual, responsive, accessibility, or functional purpose.

3. Deploy to Vercel

Create a Vercel Preview deployment for mathlog.

Configure only what is required, including the root directory, framework, build settings, output directory, and environment variables.

Do not expose secrets or commit local environment files.

Record the Preview URL.

4. Test with Playwright

Run Playwright against the deployed Preview URL.

Test the important pages and interactions, including:

* The homepage and main routes load.
* Navigation links work.
* Buttons, forms, and interactive elements work.
* Images, icons, and fonts load.
* Nested routes work after a direct refresh.
* There are no significant console errors or failed network requests.
* There is no unintended horizontal scrolling.

Test at these viewport sizes:

* Mobile: 390 × 844.
* Desktop: 1440 × 900.

Use stable selectors based on roles, labels, or test IDs.

5. Capture and review screenshots

Capture full-page screenshots of the important pages on mobile and desktop.

Also capture important states such as open menus, forms, dialogs, validation errors, or expanded sections when applicable.

Open and inspect the screenshots for:

* Broken layouts.
* Overlapping or clipped content.
* Inconsistent spacing or alignment.
* Poor mobile responsiveness.
* Missing or distorted images.
* Duplicate or unnecessary content.
* Unfinished or inconsistent UI elements.

Do not assume the site looks correct just because the automated tests pass.

6. Fix and retest

Fix confirmed functional or visual issues while preserving the existing design.

After making changes:

* Rerun the project checks.
* Create a new Preview deployment.
* Rerun the Playwright tests.
* Capture and inspect new screenshots.

Repeat only as needed until there are no significant issues.

7. Deploy to Production

Once the final Preview deployment has been tested successfully, promote it to Vercel Production.

Run a brief Playwright smoke test against the Production URL to confirm that the homepage, main routes, navigation, assets, and important interactions still work.

8. Final report

Provide:

* The Preview URL.
* The Production URL.
* Tests and viewport sizes used.
* Pages and interactions tested.
* Issues found and fixed.
* Redundant code, assets, or UI removed.
* Files changed or deleted.
* Any remaining limitations.

Do not claim that something was tested unless it was actually tested. Do not claim that the site looks correct unless the screenshots were opened and reviewed.
